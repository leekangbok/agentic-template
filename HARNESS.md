# 하네스 엔지니어링 가이드

이 문서는 AI 기반 개발에서 사용할 검증 레일, 즉 하네스(harness)를 정의합니다.
핵심 목표는 "AI가 코드를 수정한 뒤, 명확한 검증 기준을 통과할 때만 다음 단계로 진행하게 만드는 것"입니다.

## 목적

- AI가 감으로 완료 판단을 하지 않도록 막기
- 코드 변경 결과를 자동 검증 가능한 형태로 만들기
- 회귀를 빠르게 발견하기
- 사람이 결과를 리뷰하기 전에 실패를 최대한 자동으로 걸러내기

## 기본 원칙

1. 코드가 바뀌면 반드시 하네스를 실행합니다.
2. 하네스가 실패하면 다음 단계로 진행하지 않습니다.
3. 테스트, 린트, 타입체크, 빌드 검증은 가능한 한 스크립트로 표준화합니다.
4. 사람이 아는 암묵지를 명령, fixture, 예시 입력으로 외부화합니다.
5. 하네스 결과는 성공/실패가 명확해야 합니다.

## 표준 검증 진입점

이 템플릿에서는 아래 스크립트를 공통 진입점으로 사용합니다.

- `scripts/check.sh`
- `scripts/check.bat`
- `scripts/check.py`

새 프로젝트를 시작하면 위 스크립트 안에 실제 검증 명령을 채웁니다.

## 권장 하네스 구성

### 최소 단계

- 단위 테스트 실행
- 통합 테스트 또는 핵심 시나리오 검증
- **API 명세(Swagger) 최신성 확인**: `swag init` 실행 후 `docs/` 변경 확인
- **환경 설정 명세 확인**: `.env.example`과 실제 `.env` 항목의 일치 여부 점검

### 프로젝트가 커지면 추가할 것

- 타입체크
- 스냅샷 또는 골든 파일 비교
- API smoke test
- CLI 입력/출력 검증
- 회귀 테스트 세트

## 완료 기준

다음 조건이 모두 만족될 때만 "완료"로 간주합니다.

1. 관련 코드 변경이 반영됨
2. 관련 문서가 갱신됨
3. 하네스가 통과함
4. 남은 리스크가 있으면 명시됨

## AI 작업 규칙

- AI는 코드 변경 후 `scripts/check.*`를 우선 실행합니다.
- 검증이 실패하면 실패 원인을 수정하고 다시 검증합니다.
- 검증을 생략해야 하는 경우 이유를 `CONTEXT.md` 또는 최종 결과에 남깁니다.
- 새 검증 명령이 생기면 `README.md`와 이 문서(`HARNESS.md`)를 즉시 갱신합니다.

## 완료 기준 (Definition of Done)

모든 작업은 아래 기준을 모두 충족해야 "완료"로 간주합니다.

1.  **코드 검증**:
    - `scripts/check.py`의 모든 하네스 명령이 성공(`Success`)해야 함
    - 새로 추가된 기능에 대한 단위 테스트 또는 통합 테스트가 포함되어야 함
2.  **문서 동기화**:
    - `CONTEXT.md`와 `TASKS.md`가 실제 상태를 정확히 반영해야 함
    - 아키텍처 변경 시 `docs/architecture.md`가 갱신되어야 함
3.  **지식 증류**:
    - 중요한 결정 사항이나 설계 이유는 `docs/references/KNOWLEDGE_BASE.md`에 기록되어야 함
4.  **리스크 명시**:
    - 해결되지 않은 문제나 잠재적 위험 요소가 `CONTEXT.md`의 리스크 섹션에 기록되어야 함
5.  **실행 계획 준수**:
    - 복잡한 작업의 경우, 승인된 `docs/exec-plans/`의 계획대로 구현되었는지 확인되어야 함

## 프로젝트 시작 후 바로 할 일

- `backend/`와 `frontend/` 각각의 테스트 및 린트 명령을 `scripts/check.py`에서 최종 확인
- 로컬 인프라 실행을 위해 `docker-compose up -d` 명령어 및 도구 설치 확인
- 실패 시 백엔드 로그(`go run backend/main.go`)와 프런트엔드 콘솔을 모두 확인하도록 가이드
