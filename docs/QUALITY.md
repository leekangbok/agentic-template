# 품질 기준

이 문서는 이 프로젝트에서 "좋은 변경"을 무엇으로 볼지 정의합니다.

## 기본 품질 기준

- 요구사항에 맞게 동작한다
- 테스트 또는 검증을 통과한다
- 문서와 실제 구현이 일치한다
- 회귀 가능성이 줄어든다
- 리뷰 가능한 크기의 변경이다

## 기술 스택별 품질 기준

### Backend (Go)
- `go fmt`가 적용되어 코드 스타일이 일관되어야 함
- `go vet`을 통해 기본적인 프로그래밍 실수가 없어야 함
- 모든 API 엔드포인트에는 Swagger 어노테이션이 포함되어야 함

### Frontend (React)
- ESLint 규칙을 통과해야 함
- TypeScript 타입을 엄격하게 정의하고 `any` 사용을 지양함
- 빌드 프로세스(`npm run build`)에 경고 없이 성공해야 함

## 검토 질문

- 이 변경은 재현 가능한 방식으로 검증되었는가
- 실패 시 원인을 추적할 수 있는가
- 다른 모듈에 숨은 영향을 주지 않는가
- 다음 세션이 쉽게 이해할 수 있는가
